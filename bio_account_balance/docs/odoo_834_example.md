## 2️⃣ SQL VIEW з Window Functions

### SQL запит (спрощений):
```sql
CREATE VIEW bio_account_move_line_view AS (
    SELECT
        aml.id,
        aml.date,
        aml.move_id,
        aml.account_id,
        aml.partner_id,
        aml.debit,
        aml.credit,
        aml.balance,

        -- 🔑 КЛЮЧОВА ЛОГІКА: Кумулятивний баланс

        -- Initial Balance = сума ДО поточного рядка
        COALESCE(
            SUM(aml.debit - aml.credit) OVER (
                PARTITION BY aml.partner_id, aml.account_id, aml.company_id
                ORDER BY aml.date, aml.id
                ROWS BETWEEN UNBOUNDED PRECEDING AND 1 PRECEDING
            ),
            0
        ) as bio_initial_balance,

        -- End Balance = сума ВКЛЮЧАЮЧИ поточний рядок
        COALESCE(
            SUM(aml.debit - aml.credit) OVER (
                PARTITION BY aml.partner_id, aml.account_id, aml.company_id
                ORDER BY aml.date, aml.id
                ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
            ),
            0
        ) as bio_end_balance

    FROM account_move_line aml
    WHERE aml.parent_state = 'posted'
      AND aml.account_id IS NOT NULL
)
```

### Пояснення Window Function:

**PARTITION BY partner_id, account_id, company_id:**
- Розділяє дані на групи (кожна комбінація партнер+рахунок = окрема група)
- Для A.T.B. + 401010 = одна група

**ORDER BY date, id:**
- Сортує рядки в групі хронологічно
- Якщо дата однакова - сортує по id

**ROWS BETWEEN UNBOUNDED PRECEDING AND 1 PRECEDING:**
- "від початку групи до попереднього рядка"
- Це і є Initial Balance (баланс ДО поточного рядка)

**ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW:**
- "від початку групи до поточного рядка (включно)"
- Це End Balance (баланс ПІСЛЯ поточного рядка)

## 4️⃣ ПРОБЛЕМА: Неправильний _order

### ❌ Якщо model._order = 'date desc, id desc':

Odoo показує рядки в ЗВОРОТНОМУ порядку:

| id  | date       | move_name        | bio_initial_balance | debit     | credit     | bio_end_balance |
|-----|------------|------------------|---------------------|-----------|------------|-----------------|
| 300 | 2026-01-23 | VB/2026/01/0127  | **-7,744.05** ❌    | 0.00      | 14,771.44  | **-22,515.49**  |
| 201 | 2026-01-08 | BCRR/2026/00045  | **-42,051.02** ❌   | 34,306.97 | 0.00       | **-7,744.05**   |
| 200 | 2026-01-08 | BCRR/2026/00045  | **-42,667.31** ❌   | 616.29    | 0.00       | **-42,051.02**  |
| 100 | 2026-01-05 | VB/2026/01/0032  | **-40,819.52** ❌   | 0.00      | 1,847.79   | **-42,667.31**  |

**Що не так?**

Дивимося на рядок 300 (01/23):
- **bio_initial_balance = -7,744.05** - це правильно для позиції №4 в хронології
- Але показується ПЕРШИМ!
- Користувач бачить: "Initial = -7,744.05, після операції стало -22,515.49"
- Але це НЕ initial balance для ПЕРІОДУ, а initial для цього КОНКРЕТНОГО рядка в середині!

Дивимося на рядок 100 (01/05):
- Показується ОСТАННІМ
- **bio_initial_balance = -40,819.52** - це СПРАВЖНІЙ initial balance періоду!
- Але користувач його не бачить, бо він внизу списку!

### 🔥 Результат:
- Баланси розраховані ПРАВИЛЬНО в SQL
- Але показані в НЕПРАВИЛЬНОМУ порядку
- Користувач бачить Initial Balance = -7,744.05 (замість -40,819.52)
- Partner Ledger показує -40,819.52 ✓
- **РІЗНИЦЯ: невідповідність!**

## 7️⃣ Формула перевірки

### Логіка балансів:
```
bio_end_balance[i] = bio_initial_balance[i] + debit[i] - credit[i]

Для послідовних рядків:
bio_initial_balance[i+1] = bio_end_balance[i]
```

### Приклад перевірки:
```
Рядок 1 (id=100):
  End = -40,819.52 + 0 - 1,847.79 = -42,667.31 ✓

Рядок 2 (id=200):
  Initial = End[рядок 1] = -42,667.31 ✓
  End = -42,667.31 + 616.29 - 0 = -42,051.02 ✓

Рядок 3 (id=201):
  Initial = End[рядок 2] = -42,051.02 ✓
  End = -42,051.02 + 34,306.97 - 0 = -7,744.05 ✓

Рядок 4 (id=300):
  Initial = End[рядок 3] = -7,744.05 ✓
  End = -7,744.05 + 0 - 14,771.44 = -22,515.49 ✓
```

**Всі формули працюють! ✓**

## 8️⃣ Ключові моменти

### ✅ Що ПРАВИЛЬНО:
1. SQL VIEW з window functions - логіка розрахунку **ПРАВИЛЬНА**
2. PARTITION BY partner_id, account_id - **ПРАВИЛЬНО**
3. Window function ORDER BY date, id - **ПРАВИЛЬНО**
4. Формула: End = Initial + Debit - Credit - **ПРАВИЛЬНО**

### ❌ Що було НЕПРАВИЛЬНО:
1. Model `_order = 'date desc, id desc'` - **КОНФЛІКТ!**
2. Window function рахує в одному порядку (ASC)
3. Але показує в іншому (DESC)
4. Initial/End баланси не відповідають позиції рядка

### ✅ РІШЕННЯ:
```python
_order = 'date, id'  # Має співпадати з window function ORDER BY
```

## 9️⃣ Як це працює в Odoo

### Window Function (PostgreSQL):
```
За кулісами PostgreSQL:
1. Бере всі рядки для (partner=A.T.B., account=401010)
2. Сортує їх: ORDER BY date, id
3. Для кожного рядка рахує SUM від початку до поточного
4. Результат: кожен рядок має свій bio_initial_balance і bio_end_balance
```

### Odoo Model:
```
Коли користувач відкриває view:
1. Odoo читає з SQL VIEW
2. Застосовує _order для сортування результатів
3. Якщо _order ≠ window function ORDER BY → ПРОБЛЕМА!
4. Баланси показані в неправильному порядку
```

### Фікс:
```python
# Тепер _order співпадає з window function
_order = 'date, id'

Результат:
1. Window function рахує: ORDER BY date, id
2. Odoo показує: ORDER BY date, id
3. ✅ СИНХРОНІЗАЦІЯ!
```

## 🎯 Висновок

### Алгоритм роботи:

1. **SQL VIEW** створює віртуальну таблицю з window functions
2. **Window function** рахує кумулятивний баланс в порядку: `date, id`
3. **Model `_order`** має співпадати з window function `ORDER BY`
4. **Tree view** показує дані в порядку model `_order`
5. **Користувач** бачить правильну хронологію з правильними балансами
