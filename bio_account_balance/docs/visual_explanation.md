## 2️⃣ Як працює WINDOW FUNCTION

### SQL код:
```sql
SUM(debit - credit) OVER (
    PARTITION BY partner_id, account_id, company_id
    ORDER BY date, id
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
)
```

### Що це означає по-людськи:

```
PARTITION BY → Групуємо по: A.T.B. + 401010 + Компанія
              (окрема група для кожного партнера+рахунку)

ORDER BY → Сортуємо хронологічно: дата, потім id

ROWS BETWEEN → Для кожного рядка:
               рахуємо суму від початку до поточного рядка
```

## 5️⃣ Технічне рішення

### Зміна в коді:

```python
# ❌ БУЛО:
class BioAccountMoveLineView(models.Model):
    _order = 'date desc, id desc'  # Зворотній порядок!
```

```python
# ✅ СТАЛО:
class BioAccountMoveLineView(models.Model):
    _order = 'date, id'  # Хронологічний порядок!
```

### Чому це важливо:

```
SQL VIEW:
  ORDER BY date, id  ← Window function рахує так

Model:
  _order = 'date, id'  ← Відображення має бути так само!

РЕЗУЛЬТАТ:
  Порядок розрахунку = Порядок відображення ✓
```

## 🎯 Підсумок для розуміння

### 1. Бухгалтерія проста:
```
Credit на 401010 = Борг ↑ (отримали послугу, винні)
Debit на 401010  = Борг ↓ (заплатили, борг зменшився)
```

### 2. Кумулятивний баланс = "що було раніше":
```
Рядок 1: Initial = історія до цього рядка
         End = Initial + поточна операція

Рядок 2: Initial = End(рядок 1)
         End = Initial + поточна операція

...і так далі хронологічно
```

### 3. Порядок критичний:
```
Якщо показати в зворотному порядку:
  → Initial/End не відповідають позиції
  → Користувач бачить неправильні цифри

Якщо показати хронологічно:
  → Перший рядок = початок періоду ✓
  → Останній рядок = кінець періоду ✓
  → Initial наступного = End попереднього ✓
```

### 4. Фікс простий:
```
_order = 'date, id'  # Має співпадати з window function
```
